# 设计决策记录

## D1: 复盘表去掉 UNIQUE(trade_date) 约束

**原需求**：`trade_date TEXT NOT NULL UNIQUE`
**调整**：去掉 UNIQUE，改为普通索引。

**理由**：期货夜盘交易员可能在同一天有日盘和夜盘两次复盘；外汇交易员可能按 session（亚盘/欧盘/美盘）分别复盘。强制一天一条会限制使用场景。

## D2: "跳过周末"逻辑简化

**原需求**：明日计划自动跳过周末
**调整**：默认取明天日期，用户可手动修改日期。

**理由**：期货有周五夜盘（算下周一交易日），外汇周日晚开盘。不同市场的"交易日"定义不同，硬编码跳过周末会误导用户。

## D3: 弱点衰减改为每日一次

**原需求**：复盘后未命中的弱点 `weight *= 0.98`
**调整**：每日首次启动时对所有弱点执行一次衰减（`weight *= 0.98`），而非每次复盘时。

**理由**：如果用户一天复盘多次（如分 session），每次都衰减会导致未命中弱点衰减过快。每日一次更合理。

## D4: pywebview 延后到 P2

**原需求**：pywebview 首选，fallback 浏览器
**调整**：P0/P1 阶段直接用默认浏览器打开。P2 打包时再考虑 pywebview。

**理由**：pywebview 在不同系统上的 WebView 引擎差异大，调试困难。先确保核心功能稳定，再包装外壳。

## D5: 飞书定时推送改为启动时检查

**原需求**：APScheduler 每日 8:30 推送
**调整**：保留 APScheduler 定时，但增加启动时检查逻辑——如果今日未推送过且已过推送时间，立即补推。

**理由**：桌面应用不保证 24h 运行。用户可能 9:00 才开机，如果只靠定时 8:30 触发，会错过当天推送。

## D6: 前端采用单页多 Tab 设计

**决策**：不做多页面路由，用 Tab 切换（计划 / 复盘 / 弱点矩阵 / 设置）。

**理由**：vanilla JS 做 SPA 路由增加复杂度但无收益。Tab 切换更简单，用户一眼看到所有功能入口。
